### Fixed

- **Draw.io edits in Compendiq now round-trip through Confluence's native viewer (#302 Gap 2).** Previously the save path in `PageViewPage.handleDrawioSave` accepted the `xml` argument but **ignored it** â€” only the rendered PNG was pushed back to Confluence as an attachment. The result: a user who edited a diagram in Compendiq and synced back saw their changes in Compendiq's viewer, but **Confluence's native draw.io plugin could no longer open the diagram for editing** (the plugin reads the `.drawio` XML sibling file, not the PNG). The PUT route now accepts an optional `xml` field in the JSON body and uploads a matching `.drawio` sibling attachment alongside the PNG (content-type `application/xml`, filename derived by swapping `.png` â†’ `.drawio` or appending `.drawio` for non-PNG names). XML uploads up to 25 MB. Failure on the sibling upload is non-fatal: the PNG still lands and the response reports success (XML failure logged as a warning); the UTF-8-inflated size check is also non-fatal (logs + skips the sibling instead of failing the request). The PUT route now sets `bodyLimit: 40 MB` so the combined 10 MB PNG (base64) + 25 MB XML payload is actually reachable at the Fastify layer. `getMimeType` now maps `.drawio` to `application/xml` so cached sibling files are served with a consistent content type. `syncDrawioAttachments` on the sync side now downloads the `.drawio` XML sibling when present so Compendiq has both halves for local editing. Gap 3 (TipTap editor-mode dirty-diagram registry) and Gap 4 (local-page attachment storage) from the #302 issue are tracked as follow-ups. Self-hosted draw.io (#302 Gap 1) was already solved in v0.3 by bundling `draw.war` v29.6.4 into the frontend nginx image â€” no action needed. (#302)
- **Draw.io diagrams edited inside the TipTap editor now survive page reload (#302 Gap 3 + Gap 4).** The editor's `DrawioDiagramNodeView.handleSave` wrote the edited PNG into the node's transient `pngDataUri` attribute, but the page-level save never flushed it to the attachment store. The serialised body_html carried the PNG inline as base64 â€” it reloaded fine in the editor but bloated the DB row by ~14 MB per edit, and the node's `src` remained null so subsequent edits inflated the row again. New `drawio-save-drain.ts` walks the editor's doc before `handleSave` serialises HTML, uploads every pending diagram (PNG + `.drawio` XML sibling, per Gap 2) to `/api/attachments/:pageId/:filename` (Confluence pages) or `/api/local-attachments/:pageId/:filename` (standalone pages, via the Gap 4 backend), and rewrites the node's attributes with the resolved server URL so the saved HTML references the attachment cheaply. (#302 Gap 3 + Gap 4)
- **SSRF allowlist stays coherent across pods.** The module-level origin `Set<string>` in `core/utils/ssrf-guard.ts` used to drift across pods: adding a Confluence URL on Pod A left Pod B's set unaware until Pod B's next sync lazy-registered the URL. Pod B would reject the very URL the customer had just configured. `bootstrapSsrfAllowlist()` (defined in `sync-service.ts` but never called â€” `index.ts` had an inline duplicate) is now the canonical entry point and is invoked inside `buildApp()` after the Redis plugin registers, so every pod in every process populates the allowlist in the same place. A new `core/services/ssrf-allowlist-bus.ts` wires a Redis pub/sub subscriber on `ssrf:allowlist:changed`; public mutations (`addAllowedBaseUrl` / `removeAllowedBaseUrl` / `replaceAllowedBaseUrls`) broadcast the change and peer pods apply it locally without re-publishing. Publish-only-on-change keeps the channel quiet during idempotent warm-up calls (e.g. `ConfluenceClient` construction). New `addAllowedBaseUrlSilent` / `removeAllowedBaseUrlSilent` exports are used by bootstrap paths that should not flood the channel. Fails soft when Redis is unavailable (single-pod mode â€” local state still works). (#306)

- **Embedding model switch no longer caps at 2000 dimensions.** `POST /api/admin/embedding/reembed` with `newDimensions > 2000` (e.g. switching to `qwen3-embedding:4b` at 2560 dims or `qwen3-embedding:8b` at 4096) previously rolled back the whole transaction with `column cannot have more than 2000 dimensions for hnsw index`. The re-embed path now tiers the column type + index choice: `â‰¤ 2000` keeps the vector+HNSW path unchanged; `2001â€“4000` uses `halfvec(n)` + HNSW `halfvec_cosine_ops` (float16 storage, ~equivalent recall); `> 4000` stores as `vector(n)` with no HNSW index (sequential scan â€” correct but slower on large KBs, logged as a warning). DDL order now mirrors migration 048: DROP INDEX â†’ ALTER COLUMN TYPE â†’ CREATE INDEX. HNSW tuning `WITH (m=16, ef_construction=200)` applied consistently. Zod upper bound widened 8192 â†’ 16000 (pgvector's max).
